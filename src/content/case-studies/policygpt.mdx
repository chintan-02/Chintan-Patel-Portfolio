## Problem

Organizations store important policy and compliance information inside long PDF documents. Employees need quick, grounded answers — but manually searching through documents is slow, and the cost of a wrong answer in a compliance context is high.

Generic chatbots can respond quickly, but they may hallucinate or provide unsupported answers. For enterprise document Q&A, answers must be grounded, traceable, and connected back to source documents. A system that confidently invents a policy answer is worse than one that says it doesn't know.

## Solution

PolicyGPT allows users to upload policy PDFs and ask natural-language questions through a Streamlit UI backed by FastAPI. The backend extracts PDF text, cleans it, chunks it with page-level metadata, creates embeddings, and stores them in ChromaDB. For each question, it retrieves evidence, scores retrieval confidence, and only allows the LLM to generate an answer if the evidence clears a threshold — otherwise it returns a documented fallback response, not a guess.

## Key Features

- PDF upload and ingestion with PyMuPDF text extraction and cleaning.
- Metadata-rich document chunking with page-level citation support.
- SentenceTransformers embeddings (all-MiniLM-L6-v2) stored in ChromaDB.
- Semantic retrieval pipeline with per-chunk retrieval score calculation.
- Evidence-gate scoring: retrieval confidence evaluated before generation is allowed.
- Citation-backed answer generation — every supported answer is tied to specific pages.
- Confidence score surfaced alongside every answer, not hidden.
- Unsupported-question fallback: when evidence doesn't clear the threshold, the system declines rather than guessing.
- Provider-agnostic LLM layer — built against Groq Llama-3.3-70B, switchable to OpenAI-compatible endpoint.
- Evidence Explorer page for retrieval transparency — users can inspect what was retrieved and at what score.
- Working FastAPI backend with Swagger API documentation and health endpoint.
- Streamlit enterprise dashboard as the user-facing interface.
- Citation cards showing document name, page number, section title, excerpt, and retrieval score.

## Evidence-First Design

The key architectural decision is that retrieval quality *gates* generation — the LLM is not given free rein to respond to every query. Most basic RAG demos retrieve some passages and let the model do its best with whatever comes back, even when nothing relevant was actually found. PolicyGPT scores retrieval first: if evidence doesn't clear the confidence bar, the LLM call is either skipped or its output is overridden by the fallback path, and the user is told explicitly why.

For HR and compliance use cases, "I don't have enough information to answer that" is a better outcome than a fluent, wrong answer that someone might act on.

## Production Concerns

Going from Phase 1 MVP to something deployable requires: authentication and document-level access control (HR documents are sensitive by nature), persistent vector storage replacing the local ChromaDB instance, a labeled evaluation dataset to measure retrieval precision/recall and citation accuracy quantitatively rather than by spot check, request logging and monitoring for both latency and answer quality, and containerized deployment via Docker Compose.

## Roadmap

- Cloud deployment (Azure or similar) with persistent vector store.
- Retrieval and answer logging for every query.
- Custom RAG evaluation dataset and evaluation runner.
- Improved confidence scoring with calibrated thresholds.
- RAG evaluation dashboard.
- PostgreSQL metadata layer for production-grade persistence.
- Docker Compose for reproducible local development.
- Multi-document comparison across more than one policy PDF.
- Compliance summary report generation.
- Second LLM provider behind the same evidence gate to confirm provider independence.
