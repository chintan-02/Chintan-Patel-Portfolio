## Problem

Emergency intake decisions need to be fast, consistent, and safety-aware. In real clinical settings, patient acuity depends on structured information: age, chief complaint, vitals, arrival mode, pain score, and clinical context.

A machine learning model can help identify patterns in structured intake data, but healthcare AI must be designed carefully. Predictions need to be explainable, reviewable, auditable, and supervised by clinicians. A model that just outputs a number without context is not appropriate here.

## Solution

TriageAI combines ML prediction with a human-in-the-loop clinical review workflow. The backend generates an ESI 3/4/5 model prediction with confidence and probability scores. A safety-rule layer checks for higher-risk signals and can escalate the final ESI recommendation. The frontend presents the result with model confidence, class probabilities, safety flags, clinical explanation, and a recommended review action.

Clinicians can accept the recommendation or override the final ESI when additional clinical context justifies a different decision. Every decision — including overrides — is stored with a required reason, timestamp, and full audit trail.

The model is intentionally scoped to ESI 3/4/5 prediction, while higher-risk escalation is handled by explicit safety rules and clinician review rather than unsupported model claims.

## Key Features

- Structured patient intake workflow covering vitals, chief complaint, arrival mode, and clinical context.
- Real LightGBM V2 model inference for ESI 3, ESI 4, and ESI 5 acuity prediction.
- Probability and confidence output per class — not just a label.
- Safety-rule escalation logic that can raise the ESI recommendation when critical signals are present.
- Clinician accept/override workflow — clinicians remain the final authority on every case.
- Required override reason — every override is documented and stored, not silent.
- Audit trail for every prediction and every review decision.
- Dashboard with assessment history, review status, and filtering.
- Assessment detail page with persisted model and review data per case.
- PDF decision-support report generation for structured documentation.
- Local demo database reset and seed workflow with pre-built recruiter-ready demo cases.
- FastAPI health and readiness endpoints, SQLAlchemy ORM, SQLite local DB with PostgreSQL-compatible design.
- pytest-based backend and workflow validation.

## Machine Learning Scope

The deployed model is a **LightGBM V2 weighted-threshold model** trained specifically for ESI 3, 4, and 5 classification. It does not independently predict ESI 1 or ESI 2 — higher-acuity outcomes are handled through the safety-rule escalation layer and clinician review. This design keeps the ML scope precise and the safety behavior explicit rather than mixing model confidence on rare, high-severity classes with the escalation logic.

The model was selected after a bake-off against XGBoost, Random Forest, a Dense Neural Network, an RNN, and a GRU — evaluated on weighted F1 and ESI 3→5 safety-critical error rate, not raw accuracy alone.

## Human-in-the-Loop Design

TriageAI does not present the model as final clinical authority. The model generates a structured decision-support recommendation, but every clinician can accept or override it. When an override occurs, the system records the clinician's decision, their final ESI, override reason, review note, timestamp, clinician ID, and an audit trail event.

This means every decision is accountable — not just the model's recommendation, but the human decision on top of it.
